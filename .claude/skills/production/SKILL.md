# 생산 라인(Production Line) 구현 가이드

이 스킬은 "반도체 시료 생산주문관리 시스템"에서 **생산 라인(ProductionLine)** 로직을 구현하거나 수정할 때
따라야 할 규칙을 정리한다. 작업 전 저장소 루트의 `CLAUDE.md`를 먼저 확인해 전체 MVC 구조와 상태 흐름을
파악한다.

## 핵심 원칙: 전역 단일 생산 슬롯

**생산 라인은 제품(시료)에 상관없이 전체 시스템에서 단 하나의 활성 생산 job만 가질 수 있다.**

이 시스템의 "생산 라인"은 물리적으로 하나이므로, 서로 다른 시료의 주문이라도 절대 동시에 생산될 수 없다.
하나의 job이 완료되어야 다음 job(같은 시료든 다른 시료든 상관없이 전역 FIFO 순서상 다음 것)이 시작된다.

예시: 시료 A의 주문이 현재 생산 중이라고 하자. 이때 시료 B의 새 주문이 승인되어 재고 부족으로 `PRODUCING`
판정을 받더라도, 시료 B의 생산은 즉시 시작되지 않는다. 설령 B에 필요한 원료/설비가 이론상 A와 전혀
무관하더라도, 전역 큐에서 A가 끝날 때까지 B는 대기해야 한다. "시료별로 독립된 생산 라인이 있다"는 가정은
틀렸다 — 생산 라인은 오직 하나뿐이다.

## FIFO 대기열 구조

- 시료별로 별도 큐를 두지 않는다. 전체 시스템에 **단 하나의 FIFO 큐**만 존재하며, 그 큐는 현재 진행 중인
  job 1개와 대기 중인 job들로 구성된다.
- 새 job이 도착했을 때:
  - 현재 진행 중인 job이 없으면 (큐가 비어 있으면) 즉시 시작한다.
  - 현재 진행 중인 job이 있으면, 새 job의 sampleId가 무엇이든 상관없이 큐 맨 뒤에 추가하고 대기시킨다.
    이때 대기 중인 job의 시작/종료 예정 시각은 Enqueue 시점에 즉시 계산되어 확정된다(이전 항목 — 대기열
    마지막 항목, 없으면 현재 진행 중인 job — 의 종료 예정 시각을 그대로 새 job의 시작 시각으로 삼는다).
    이렇게 해야 프로그램이 오랫동안 꺼져있어도 대기열 전체가 wall-clock 기준으로 이론상 계속 진행된 것으로
    정확히 계산된다.
- 하나의 job이 완료되면 큐에서 제거하고, 다음 job이 있으면 대기열에 있던(이미 Enqueue 시점에 확정된) 시작/
  종료 예정 시각을 그대로 사용해 현재 job으로 승격한다(시각을 다시 계산하지 않는다).
- 도착 순서(FIFO)는 시료 종류와 무관하게 그대로 보존되어야 한다 — 예: A, B, A, C 순서로 job이 등록되었다면
  반드시 A → B → A → C 순서로 하나씩 처리된다(같은 시료의 job이라고 순서를 앞당기거나 묶어 처리하지 않는다).

## 계산식

- 실 생산량 = `ceil(shortage / yieldRate)` (`shortage`는 이 job 등록 시점의 부족분, `yieldRate`는 해당 시료의
  수율)
- 총 생산 시간(분) = `avgProductionTime(분, 소수점 가능) * 실 생산량`
- 완료 판정은 **wall-clock(실제 경과 시간)** 기준이다. job 시작 시각과 예상 완료 시각(시작 시각 + 총 생산
  시간)을 기록해두고, 이후 어느 시점에 조회하든(프로그램이 종료되었다가 재실행되어도) "현재 시각이 예상 완료
  시각을 지났는가"로 완료 여부를 판정한다. 실행 여부가 아니라 실제로 흐른 시간만큼 생산이 진행된 것으로
  취급한다.

## 앱 종료 중에도 대기열이 계속 진행되어야 함 (중요 버그 수정 이력)

과거에는 대기열(`pendingQueue_`)에 들어가는 job의 시작/종료 예정 시각이 그 job이 실제로 "현재 job"으로
승격되는 순간(`ProcessCompletions` 호출 시점의 `now()`)에야 정해졌다. 이 때문에 프로그램이 오랫동안
꺼져있으면, 앞선 job이 이미 (wall-clock 기준으로) 끝났어야 할 시각이 지나도 그 사실을 프로그램이 "실행
중에 목격"하지 못하는 한 뒤따르는 job들의 시계가 전혀 진행되지 않는 버그가 있었다.

이를 수정해 `Enqueue` 시점에 대기열에 들어가는 job의 시작 시각을 즉시 계산해서 확정하도록 변경했다.

- 큐가 완전히 비어있으면: `startedAt = now()` (즉시 시작)
- `currentJob_`은 있지만 대기열이 비어있으면: 새 job의 `startedAt = currentJob_->expectedEndAt` (현재
  job이 끝나는 바로 그 시각)
- 대기열에 이미 항목이 있으면: 새 job의 `startedAt = pendingQueue_.back().expectedEndAt` (대기열 맨 뒤
  항목이 끝나는 바로 그 시각)
- 어느 경우든 `expectedEndAt = startedAt + 총생산시간`으로 즉시 계산해서 확정한다.

이렇게 대기열의 모든 job이 등록 시점에 이미 절대적인(absolute) 시작/종료 시각을 갖게 되므로,
`ProcessCompletions`는 job을 승격시킬 때 시각을 재계산할 필요 없이 그대로 사용하면 된다. 이 원칙 덕분에
앱 재시작 시 `ProcessCompletions`를 한 번만 호출해도, 그 사이 실제로 경과한 시간만큼 여러 job이 한꺼번에
연쇄 완료 처리될 수 있다.

## 승인 로직과의 연계

`Order` 승인 시점에 재고 부족 여부(`CONFIRMED`/`PRODUCING` 분기)를 판정하려면, "이미 이 시료에 대한 생산
job이 (현재 진행 중이든 대기 중이든) 존재하는지"를 먼저 확인해야 한다.

- 이 확인은 **전역 큐 전체**(현재 진행 중인 job + 대기 중인 job들)에서 해당 `sampleId`를 가진 job이 하나라도
  있는지를 검사하는 방식이어야 한다. 전역 단일 슬롯 구조에서는 이 시료의 job이 "지금 당장 생산 중"이 아니라
  "대기열 어딘가에" 있을 수도 있기 때문에, 단순히 "지금 이 시료가 현재 job인지"만 확인하면 안 된다.
- 이 시료에 대한 job이 이미 존재한다면(현재든 대기든), 해당 시료의 가용 재고를 사실상 0으로 취급한다 — 즉
  이미 생산 대기 중인 물량이 재고를 채워줄 예정이므로, 신규 주문은 재고 여부와 무관하게 무조건 `PRODUCING`으로
  보내고 전역 대기열 맨 뒤에 추가해야 한다.

## 계층별 책임 분리

- **Produce** (`produce_agent`가 담당): 전역 단일 FIFO 큐(현재 job 1개 + 대기열) 구조 소유, 실 생산량/총
  생산 시간 계산, wall-clock 기준 완료 판정, 완료 시 Model API(`Order::CompleteProduction`) 호출. "이 시료의
  job이 이미 큐에 존재하는가"를 전역 큐 전체에서 조회하는 API를 제공한다. Sample/Order 클래스 정의나 콘솔
  입출력에는 관여하지 않는다.
- **Controller** (`controller_agent`가 담당): 주문 승인 시 produce 계층의 "이 시료의 job이 이미 존재하는지"
  조회 API를 호출해 가용 재고 계산에 반영(존재하면 가용 재고 0으로 취급)한다. "생산 라인 조회" 메뉴에서는
  전역 단일 현재 job과 전역 대기열을 그대로 조회해 View에 전달한다(시료별로 나누어 조회하지 않는다).
- **View** (`view_agent`가 담당): Controller가 전달한 데이터를 화면에 표시만 한다. 전역 큐 구조이므로 더 이상
  시료별로 나눠 보여줄 필요가 없다 — "현재 생산 중: job 1개" + "대기열: N개(여러 시료가 섞여 있을 수 있음)"
  형태로 단순화해서 표시한다.
- **Model** (`model_agent`가 담당): Sample/Order 클래스 정의, 상태 전이 규칙, 재고 증감 메서드 자체.
  produce_agent는 이 API를 호출만 한다.
- **Data** (`data_agent`가 담당): 생산 시작/완료 예정 시각을 포함한 생산 상태를 JSON으로 영속화한다. 저장
  스키마는 여전히 job(주문) 단위 목록이면 되며, "전역 단일 큐"라는 사실은 순서(FIFO 순서)로 표현된다 —
  별도의 시료별 필드 구분 없이 하나의 순서 있는 목록으로 저장/복원한다.

## 작업 시 체크리스트

1. `CLAUDE.md`의 생산 라인 규칙(FIFO, 실 생산량/총 생산 시간 계산식, wall-clock 완료 판정)과 모순되지 않는지
   확인한다.
2. 큐 자료구조가 시료별(`sampleId` 키를 가진 map/큐 모음)이 아니라 **전역 단일 큐 하나**로 되어 있는지
   확인한다. 시료별로 나뉜 큐가 남아 있다면 이는 잘못된 설계다.
3. 새 주문이 승인될 때, "이 시료의 job이 이미 전역 큐(현재 job + 대기열) 어딘가에 있는지" 검사를 빠뜨리지
   않았는지 확인한다 — 현재 job만 확인하고 대기열을 빠뜨리면 안 된다.
4. 위 검사에서 이미 job이 존재하는 시료는 가용 재고를 0으로 취급해 신규 주문을 무조건 `PRODUCING`으로
   보내고 전역 대기열 맨 뒤에 추가하는지 확인한다.
5. "생산 라인 조회" 메뉴/API가 시료별이 아니라 전역 단일 현재 job 1개 + 전역 대기열을 반환하는지 확인한다.
6. FIFO 순서가 시료 종류에 상관없이 순수하게 "도착(등록) 순서"를 보존하는지 확인한다(서로 다른 시료의 job이
   섞여 있어도 순서가 뒤바뀌지 않아야 한다).
7. 실 생산량 = `ceil(shortage / yieldRate)`, 총 생산 시간 = `avgProductionTime * 실 생산량`이 정확히
   반영되었는지 확인한다.
8. 완료 판정이 프로그램 실행 시간이 아니라 실제 경과 시간(wall-clock, `system_clock` 기준)으로 이루어지는지,
   재실행 시에도 저장된 시작/완료 예정 시각을 기준으로 즉시 완료 처리되는지 확인한다.
9. 대기열에 새로 들어가는 job의 시작/종료 예정 시각이 Enqueue 시점에 즉시(이전 항목의 종료 시각을 이어받아)
   계산되는지 확인한다 — job이 실제로 현재 job으로 승격되는 순간까지 시각 계산을 미루면 앱이 꺼져있는 동안
   시계가 흐르지 않는 버그가 재발한다.
10. 새 소스 파일 추가 시 `SampleOrderSystem/SampleOrderSystem.vcxproj`에 `ClCompile`/`ClInclude` 항목을
    등록한다.
11. 생산 시작/완료 예정 시각 저장 스키마를 바꾸는 경우 `data_agent`, `model_agent`와 필드 정의를 조율한다.
